3.1. Observations on Standard Library¶

  1. Marcos use define and definelong which are deprecated per below

  2. DRY (Don’t Repeat Yourself). The macros are repeated

    1. within a file -repeat the macro to add each new parameter

    2. across files - the macro content is largely the same - but the macro name changes to specify the icon to use

  3. Close Coupling

    1. Macros are defined in the icon file for each icon

  4. Lack of consistency

    1. icon sizes

      1. some have multiple sizes

      2. for those that have one size, it is not standard/common with the other stdlib libraries

    2. icon files

      1. some break up the icons into categories sub-directories, some don’t

      2. some have an `all.puml` file that includes all sprite content per category, some don’target

  5. There’s no way to scale the icons from the macros (the !define statements) except for Material. Plantuml does support scaling per https://forum.plantuml.net/4267/scaling-of-the-sprites-or-images. Support for scaling stereotypes is available since March 2019.

  6. At least 3 parameters are required and are rendered in the diagams `!define Batch(e\_alias, e\_label, e\_techn)`.

    1. In other words, the marcos force a user to specify things they may not care about e.g. to specify color in this macro, the user needs to specify the preceeding parameters also

      `!define WHATEVER(_alias, _label, _shape, _color) ENTITY(_shape,_color,whatever,_label, _alias,WHATEVER)`

    2. So if we wanted a colored or scaled icon only (with no label or technology), we have to implement it ourselves.

  7. Some libaries have an `all.puml` that includes all icons and their macros for a given category/subset of icons. This allows including all icons by including one file - rather than an include per icon.

  8. The superset of parameters provided by the macros are

    1. sprite

    2. color

    3. scale (material library only)

    4. shape (Type 1 (e_type), 4 (_shape))

    5. technology

    6. description

    7. label

Overall, the PlantUML Stdlib lacks consistency and therefore user friendliness. We need to change this to empower the user with PlantUML.

3.2. What a User Wants In A Standard Library¶

  1. Consistency of macros, files, sizes

  2. Specify the things they care about

    1. The current superset of parameters provided by the existing macros should be supported

      1. sprite

      2. color

      3. scale (material library only)

      4. shape (Type 1 (e_type), 4 (_shape))

      5. technology

      6. description

      7. label

    2. Default values specified when the icons are built, but the ones the user cares about can be overridden by the user in their diagram file (at runtime)

  3. Decoupling of the sprite (the image), from the decoration (text, shape, color, scale etc…)

    1. To benefit from other people’s work e.g. if a new parameter or decoration/theme is added they like, they want to use it in their diagrams

  4. Grouping of icons into categories.

    1. A user can include a file for that category that incldues all the icons (and not have to include a file for every icon)

  5. Future Proofing

    1. uses supported PlantUML features, and can be easily changed in the future.

  6. To be able to easily contribute their icon set to stdlib
