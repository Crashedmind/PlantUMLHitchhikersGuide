3. Standard Library¶

3.1. Observations on Standard Library¶

  1. Marcos use define and definelong which are deprecated per below

  2. DRY Do not Repeat Yourself. The macros are repeated

    1. within a file -repeat the macro to add each new parameter

    2. across files - the macro content is largely the same - but the macro name changes to specify the icon to use

  3. Close Coupling

    1. Macros are defined in the icon file for each icon

  4. Lack of consistency

    1. icon sizes

      1. some have multiple sizes

      2. for those that have one size, it is not standard/common with the other stdlib libraries

    2. icon files

      1. some break up the icons into categories sub-directories, some don’t

      2. some have an `all.puml` file that includes all sprite content per category, some don’target

  5. There’s no way to scale the icons from the macros (the !define statements). Plantuml does support scaling per https://forum.plantuml.net/4267/scaling-of-the-sprites-or-images. Support for scaling stereotypes is available since March 2019.

  6. At least 3 parameters are required and are rendered in the diagams “!define Batch(e_alias, e_label, e_techn)”.

    1. So if we wanted a colored or scaled icon only (with no label or technology), the only option is to use the raw sprite (more on that later).

Overall, the PlantUML Stdlib lacks consistency and therefore user friendliness. We need to change this to empower the user with PlantUML.

3.2. What a User Wants In A Standard Library¶

  1. Consistency of macros, files, sizes

  2. Specify the things they care about

    1. The marcos force a user to specify things they may not care about e.g. to specify color in this macro, the user needs to specify the preceeding parameters also

      `!define WHATEVER(_alias, _label, _shape, _color) ENTITY(_shape,_color,whatever,_label, _alias,WHATEVER)`

    2. Default values specified when the icons are built, but can be overridden by the user in their diagram file (at runtime)

  3. Decoupling of the sprite (the image), from the decoration (text, shape, color, scale etc…)

  4. Future Proofing

    1. uses supported PlantUML features, and can be easily changed in the future.
